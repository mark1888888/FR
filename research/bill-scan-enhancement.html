掃描帳單強化方案

在 OCR 掃描前加入「信用卡選擇器」，以解決多卡用戶記帳錯配、匯率錯算、餘額錯扣的問題。 本文件包含 UX 互動原型、狀態機設計、資料模型、React & Vanilla JS 實作、OCR 強化策略與完整 Rollout 計畫。

Web App · React / Vue Tesseract.js 5.x Supabase Enhancement Proposal Prototype Ready

1 · 摘要與 TL;DR

多卡錯配率
約 34%
模擬 3 卡用戶、現有流程下
OCR 辨識率
+18%
加入「銀行 hint」後預估提升
首次操作時間
-40%
預選卡比掃完再逐筆修正更快
實作成本
~3 天
含 UI + 狀態機 + DB 遷移

本研究針對 FlowRich 既有的 openBillScanner() 流程 (見 js/app.js:2946-3570) 提出升級： 在 OCR 之前插入一個「信用卡選擇器 Step」，把使用者在掃描後「逐筆手動指派帳戶」的操作 提前到掃描前做一次選擇，並讓該選擇同時驅動 (a) 解析結果預填、(b) OCR 針對該銀行帳單模板的 hint、 (c) 餘額計算幣別與循環日期邏輯。

核心改動只有三件事：
① 新增一個「SelectCard」掃描前畫面，帶搜尋/搜最近用過。
② 把 accountId 從「解析後再填」移到「掃描前決定」，解析結果直接全部帶入。
③ 擴充 accounts 資料模型，加入 bankCode / billingCycleDay / ocrHints 三個欄位， 讓下游邏輯更聰明。

3 · 為何要「掃描前」選卡而非「掃描後」

現行：掃描後選卡

  • 使用者心智負擔最重發生在「看到一堆行」之後
  • 每行都是獨立下拉，多卡合併時必須重複選
  • 無法為 OCR 提供 hint
  • 錯了要回頭改餘額

提案：掃描前選卡

  • 一次選擇、全部套用 (符合任務單位: 一張帳單 = 一張卡)
  • 可把選到的卡資料餵進 OCR 當成先驗 (銀行名、卡號末四碼、幣別)
  • 可預判帳單週期 → 自動填入扣款日
  • 心智模型符合「我現在要對一張卡做記帳」的自然流程

3.1 為什麼「一張帳單 = 一張卡」幾乎必定成立

在台灣發行的信用卡帳單中，單一 PDF/紙本帳單對應一個發卡銀行下的單一卡號。 即便是聯名卡、家庭附卡，也會在同一張帳單上共用主卡帳戶 — 這對記帳應用來說， 支出應該記錄在主卡帳戶。所以不存在「一張帳單跨兩個 FlowRich 帳戶」的情境。

唯一例外：使用者拿了兩張不同帳單的照片拼在一起上傳。這屬於使用者錯誤使用， 我們在 UI 上會用「您正在為 O O 卡掃描帳單」作為持續提示避免誤會。

5 · 強化後技術架構

新增 CardPicker、billScanMachine、bill_scans 表、bank-aware parser、dedupe cache。

擴充 accounts 欄位、imagePreprocess 吃 bankCode 參數、billParser 依銀行切換 regex 組。

保留 既有 Tesseract worker、_cleanOcrText、_preprocessImage 的核心邏輯 (Otsu 二值化等)。

5.1 為什麼要抽 ScanStore / state machine

現行 _billParsedItems 與 DOM 狀態耦合過深 (見 openBillScanner/resetBillScanner/processBillImage 三處都在直接操作 document)。 掃描流程現在變成 4 個明確 stage (idle → selectingCard → uploading → ocr → reviewing)， 有明確轉移條件與副作用 — 這是 state machine 的典型使用場景，讓 UI 與邏輯解耦。

6 · 狀態機設計

關鍵設計決定

  • CARD_PICKED 的 payload 會 cache accountId / bankCode / currency，後續所有 state 都只讀這包 context。
  • CHANGE_CARD 從 uploading → selectingCard 不會清掉圖片 (如果已選)，降低回退成本。
  • reviewing 是唯一允許使用者逐筆修改 accountId 的狀態，預設仍是全部指派到選定的卡 (極少用戶會想拆)。
  • saved 是 terminal — 觸發 bill_scans 表寫入一筆 scan 紀錄以供未來去重。

可用現成工具

推薦 @xstate/react (成熟 DevTools + 視覺化) 或 zustand + 手寫 reducer (輕量 ~1KB)。 若只是為了這個流程，推薦 Zustand 方案，避免引入大型 state machine lib。

7 · 資料模型 & DB 遷移

7.1 accounts 擴充欄位

現行 accounts 只有 id / name / type / balance / currency。我們加入卡片相關的 metadata：

7.2 新增 bill_scans 表

紀錄每次掃描的 metadata，用來 (a) 去重、(b) 個人化 Recent、(c) 未來做精準度回饋。

CREATE TABLE bill_scans (
  id                UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID         NOT NULL REFERENCES auth.users(id),
  account_id        UUID         NOT NULL REFERENCES accounts(id),
  image_sha256      TEXT         NOT NULL,  -- 整張圖片的 hash，去重用
  statement_period  DATERANGE,                -- 帳單週期
  raw_text          TEXT,                     -- OCR raw，除錯/重解析
  parsed_count      INT          NOT NULL DEFAULT 0,
  saved_count       INT          NOT NULL DEFAULT 0,
  ocr_confidence    NUMERIC(4,3),              -- 0.000 ~ 1.000
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
  UNIQUE(user_id, image_sha256)         -- 同一張圖同一人不可重掃
);

-- 在 expenses 加一個來源指向
ALTER TABLE expenses ADD COLUMN bill_scan_id UUID REFERENCES bill_scans(id);

8.1 介面定義 (port / adapter)

// services/billScanService.ts
export interface BillScanService {
  listCreditCards(opts?: { sortBy?: 'recent' | 'name' }): Promise<CreditAccount[]>;

  preprocessImage(file: File, hints?: { bankCode?: BankCode }): Promise<ImageBitmap>;

  ocr(image: ImageBitmap, hints?: {
    bankCode?: BankCode;
    lang?: string;           // default chi_tra+eng
    psm?: number;             // default 6
    whitelist?: string;       // 限縮字元集
  }): Promise<{ text: string; confidence: number }>;

  parse(rawText: string, card: CreditAccount): ParsedBillItem[];

  dedupe(items: ParsedBillItem[], card: CreditAccount): Promise<ParsedBillItem[]>;

  commit(items: ParsedBillItem[], scanId: string): Promise<{ savedCount: number }>;
}

每一個 method 都是 pure / 有明確 IO，方便單元測試與 mock。OCR / Parse 不直接吃 accountId 而是吃 card 物件，避免下游還要去查一次。

9 · React 實作

9.1 ScanStore (Zustand)

// stores/scanStore.ts
import { create } from 'zustand';

type Stage = 'idle' | 'selectingCard' | 'uploading' | 'preprocessing' | 'ocr' | 'reviewing' | 'saved' | 'error';

export const useScanStore = create<{
  stage: Stage;
  ctx: ScanContext;
  open: () => void;
  pickCard: (card: CreditAccount) => void;
  changeCard: () => void;
  setImage: (dataUrl: string, hash: string) => void;
  setItems: (items: ParsedBillItem[], conf: number) => void;
  fail: (msg: string) => void;
  reset: () => void;
}>((set) => ({
  stage: 'idle',
  ctx: { card: null, imageDataUrl: null, imageHash: null, rawText: null, items: [], error: null, ocrConfidence: null },
  open: () => set({ stage: 'selectingCard' }),
  pickCard: (card) => set((s) => ({ stage: 'uploading', ctx: { ...s.ctx, card } })),
  changeCard: () => set({ stage: 'selectingCard' }),  // 保留圖片!
  setImage: (u, h) => set((s) => ({ stage: 'preprocessing', ctx: { ...s.ctx, imageDataUrl: u, imageHash: h } })),
  setItems: (items, conf) => set((s) => ({ stage: 'reviewing', ctx: { ...s.ctx, items, ocrConfidence: conf } })),
  fail: (msg) => set((s) => ({ stage: 'error', ctx: { ...s.ctx, error: msg } })),
  reset: () => set({ stage: 'idle', ctx: { card: null, imageDataUrl: null, imageHash: null, rawText: null, items: [], error: null, ocrConfidence: null } }),
}));

9.2 CardPicker 元件

// components/scan/CardPicker.tsx
import { useEffect, useMemo, useState } from 'react';
import { useScanStore } from '@/stores/scanStore';
import { billScan } from '@/services/billScanService';

export function CardPicker() {
  const pickCard = useScanStore((s) => s.pickCard);
  const [cards, setCards] = useState<CreditAccount[]>([]);
  const [q, setQ] = useState('');
  const [selected, setSelected] = useState<CreditAccount | null>(null);

  useEffect(() => {
    billScan.listCreditCards({ sortBy: 'recent' }).then(setCards);
  }, []);

  const { recent, others } = useMemo(() => {
    const filtered = cards.filter((c) =>
      !q || c.name.toLowerCase().includes(q.toLowerCase()) || c.last4?.includes(q));
    const sevenDays = Date.now() - 7 * 86_400_000;
    return {
      recent: filtered.filter((c) => c.lastScannedAt && new Date(c.lastScannedAt).getTime() > sevenDays),
      others: filtered.filter((c) => !c.lastScannedAt || new Date(c.lastScannedAt).getTime() <= sevenDays),
    };
  }, [cards, q]);

  return (
    <div className="card-picker-page">
      <SearchInput value={q} onChange={setQ} placeholder="搜尋卡名或末四碼" />

      {recent.length > 0 && (
        <Section title="最近使用">
          {recent.map((c) => (
            <CardRow key={c.id} card={c} active={selected?.id === c.id}
              onPick={() => setSelected(c)} />
          ))}
        </Section>
      )}

      <Section title="全部卡片">
        {others.map((c) => (
          <CardRow key={c.id} card={c} active={selected?.id === c.id}
            onPick={() => setSelected(c)} />
        ))}
      </Section>

      <PrimaryButton disabled={!selected} onClick={() => pickCard(selected!)}>
        下一步 · 上傳帳單
      </PrimaryButton>
    </div>
  );
}

9.3 useBillScan — 驅動整條流水線

// hooks/useBillScan.ts
export function useBillScan() {
  const { ctx, setImage, setItems, fail } = useScanStore();

  const run = useCallback(async (file: File) => {
    if (!ctx.card) throw new Error('請先選擇信用卡');

    try {
      const hash = await sha256OfFile(file);
      const dataUrl = await fileToDataUrl(file);
      setImage(dataUrl, hash);

      const bmp = await billScan.preprocessImage(file, { bankCode: ctx.card.bankCode });
      const { text, confidence } = await billScan.ocr(bmp, { bankCode: ctx.card.bankCode });
      const items = billScan.parse(text, ctx.card);
      const filtered = await billScan.dedupe(items, ctx.card);

      setItems(filtered, confidence);
    } catch (e: any) {
      fail(e.message ?? '掃描失敗');
    }
  }, [ctx.card]);

  return { run };
}

function _renderCardPicker() {
  var d = U();

12 · 進階：自動偵測信用卡 (可選功能)

使用者點了「掃描」但很懶不想選卡怎麼辦？我們可以讓系統嘗試自動猜，但以「預先選擇」為主要流程 — 自動辨識只是錦上添花。

12.1 三種訊號可用

  1. Logo / 銀行名稱文字 — OCR 掃第一區塊，搜尋「玉山」「國泰世華」「中信銀行」等。命中率約 70%。
  2. 末四碼 — 帳單一定會印 card number 的末四碼。若使用者在 accounts.last4 有填，直接 regex 匹配。命中率約 95%（當 last4 已設定）。
  3. 帳單週期日期 — 帳單上「結帳日 2026/04/15」與 accounts.statementDay 比對。輔助訊號。

12.2 建議實作

async function autoDetectCard(imageFile, candidates) {
  // 用較低解析度先做一次快 OCR — 只要前 20 行
  const quickText = await quickOcrHeader(imageFile);

  // Signal 1: 末四碼
  for (const c of candidates) {
    if (c.last4 && quickText.includes(c.last4)) return { card: c, confidence: 'high' };
  }

  // Signal 2: 銀行名稱關鍵字
  for (const c of candidates) {
    const bank = BANK_OCR_HINTS[c.bankCode];
    if (bank?.displayName && quickText.includes(bank.displayName)) {
      return { card: c, confidence: 'medium' };
    }
  }

  return { card: null, confidence: 'none' };
}

13 · 邊界情境 & 錯誤處理

情境現況行為強化後建議
使用者沒有任何信用卡帳戶預設取第一個 account (可能是現金!)提示「請先新增至少一張信用卡」，引導到帳戶頁
只有一張信用卡照一般流程自動跳過 CardPicker 直接進 Upload — 避免不必要點擊
外幣帳單 (如 USD)用選擇的 account 幣別先從 card.currency 決定，若帳單上偵測到 USD/JPY 文字不符則彈提示
重複上傳同一張圖全部再 parse 一次、再入帳一次SHA-256 hash 查 bill_scans，命中則提示「此帳單已匯入過」
圖片模糊／OCR 信心度低靜默返回空結果confidence < 0.5 時主動提示「建議重拍或改用 PDF 原檔」
使用者掃到一半切換卡不支援CHANGE_CARD 保留圖片，重跑 parse 只花 < 3 秒 (image 不變，rawText 也可 cache)
帳單跨月 (例如 03/25 - 04/24)逐筆日期直接用用 card.statementDay 驗證期間合理性，異常日期標紅讓使用者決定
OCR worker 載入失敗 (網路問題)顯示 error增加 retry 按鈕；worker 成功後 cache 到 IndexedDB 避免重下載

15.1 建議上線順序

強烈建議 Phase 1 先用 Vanilla JS 在既有 app.js 上做 — 因為只動流程、不改 OCR 核心，風險極低而且能立刻驗證 UX 假設。

Phase 2 的 bank-aware OCR 是投資報酬最高的部分 — 玉山/國泰/中信三家覆蓋台灣信用卡市佔約 60%， 只要把這三家的模板做好，整體辨識率就會有感提升。

Phase 5 的 React 重構可延後到有更多 feature 需求時再做 — 當前只為了這個流程做 SPA 化不划算。
